 Enumerables

The Enumerable mixin provides collection classes with several traversal and seraching methods, and with the ability to sort. The class must provide a method 'each', which yields successive members of the collection. If Enumberable#max,#min, or #sort is used, the objects in the collection must also implement a meaningful <=> operator, as these methods rely on an ordering between members of the collection.

Since arrays are very popular in Ruby programming, I'll discuss them a bit. Here are some array enumberator methods: collect/!, combination, cycle, delete_if, drop_while, each, each_index, find_index, index, keep_if, map/!, permutation, reject/!, repeated_combination, repeated_permutation, reverse_each, rindex, and select/!.

The map enumerator is particularly useful and one of my favorites.

map { |obj| block } -> array

Examples

(1..4).map { |i| i*i } #=> [1, 4, 9, 16]
(1..4).collect { "cat" } #=> ["cat", "cat", "cat", "cat"]

Source: ruby-doc.org

 Gregoric Thinking Style and Fixed vs. Growth Mindset

According to this Gregoric Thinking Style test I have taken, I am mainly a Concrete Random and Abstract Sequential thinker.

As a Concrete Random thinker, I like experimenting to find answers, take risks, use intuition, and solve problems independently. I learn best when able to use trial-and-error approaches, compete with others, and be given the opportunity to work through problems themselves. These are difficult for me: restrictions and limitations, formal reports, routines, re-doing anything once it's done, keeping detailed records, showing how I got an answer, choosing only one answer, and having no option.

As an Abstract Sequential thinker, I like to be heard, analyze situations before making a decision, and applying logic in solving or finding solutions to problems. I learn best when I have access to experts or references, am placed in stimulating environments, and am able to work alone. These are difficult for me: being forced to work with those of differing views, too little time to deal with a subject thoroughly, repeating the same tasks over and over, have lots of specific rules and regulations, "sentimental" thinking, expressing my emotion, being diplomatic when convincing others, and not monopolozing a conversation.

After having gone through almost 3 weeks at DBC, I have seen how my two thinking styles have affected my learning. While the concrete random has mostly been positive for learning HTML/CSS/JS, my abstract sequential has made a few things more difficult. Since the pace at DBC is so fast-paced, I am forced to move on from languages/topics/challenges faster than I would like, and not have the opportunity to completely learn them inside-out. Also, I often ask experts when I am uncertain, and in solo challenges, I am unable to ask friends (other engineers) for help. In order to combat this, I am forcing myself to get adjusted with DBC's pace and to trust in myself/cohorts/guides/system.

I really enjoyed learning Carol Dweck's breakdown of a Fixed vs. Growth Mindset. While I would subscribe more to the growth mindset, I definitely see myself as a fixed mindset in certain areas and at certain times. I also attribute some of those limitations to my desire for comfort. In many instances, I will avoid challenges and give up easily because I'm unwilling to be uncomfortable. That's why "being comfortable with being uncomfortable" is one of my 2014 new year's resolutions, and DBC is definitely a part of that!

 Why is JavaScript popular in web development? What are the great things about JavaScript? What are its limitations?

JavaScript is the most popular client-side language for all browsers, so it is everywhere in web development.

There are plenty of reasons that make JavaScript so popular. First, it is easy to learn, especially coming from a programming background. Second, there is a large open source community with built-out libraries for everything you can think of. Lastly, it is lightweight and fast, and works great with Java and C++.

While JavaScript is a great language, it still does have its limitations. There are concerns regarding security issues. Also, there can be rendering variations across different platforms and browsers, which makes testing difficult.

 What are the differences between relative, absolute, and fixed positioning? #DBCRAFP

First off, all elements start out as static, meaning they don't have any default values but flow naturally onto a page.

With relative positioning, you add in positioning attributes and that would only affect the element/s relative to where it would be if it was static. So if you type in left: 10px, the element would position itself 10px to the left "relative" to where it would have been without that attribute (static).

With absolute positioning, you are able to place an element exactly anywhere on the page. You give it top, right, bottom, left position attributes, and the element will take that position of the page. These values will be relative to the next parent element with relative or absolute positioning.

Lastly, with fixed positioning, the element is positioned relative to the viewport, or the browser window itself. The viewport doesn't change when the window moves, so even if a user scrolls the page, the element will stay in position.
